# PRD: VC 구주 세일즈 CRM (코드네임: SecondaryDesk)

> 만기 펀드 EXIT를 위한 구주(Secondary) 세일즈 활동을 관리하는 내부용 B2B Sales CRM
> 작성 목적: Claude Code 기반 개발을 위한 제품 요구사항 정의서

---

## 0. 이 문서를 읽는 Claude Code에게

이 PRD는 비개발자가 Claude Code와 함께 단계적으로 구현하는 것을 전제로 작성됨. 다음 원칙을 따를 것:

- **Phase 1(MVP)부터 순서대로** 구현하고, 각 Phase는 독립적으로 동작 가능해야 함.
- 데이터 모델(§4)이 이 제품의 핵심이므로 **스키마를 먼저 확정**하고 시작할 것.
- 임의로 기능을 추가하거나 스키마를 변형하지 말 것. 모호한 부분은 구현 전에 질문할 것.
- 모든 UI 텍스트와 도메인 용어는 **한국어** 기준(투자사, 조합, 구주, 드라이파우더 등).

## 1. 배경 및 문제 정의

### 1.1 비즈니스 맥락
- 담당 펀드가 만기에 도달하여 **EXIT**가 올해의 핵심 과업.
- EXIT 수단은 보유 포트폴리오사의 **구주를 후속 투자사에 세일즈**하는 것.
- 이 활동은 본질적으로 B2B Sales와 동일한 구조 — "매물(구주)"을 "매수 후보(투자사)"에게 파는 일.

### 1.2 현재의 문제 (As-Is)
| 문제 | 영향 |
|---|---|
| 개별 심사역이 각자 네트워크에 의존해 산발적으로 진행 | 활동이 휘발됨 |
| 컨택 이력이 기록·축적되지 않음 | 같은 투자사에 중복 컨택, 학습 부재 |
| 구성원 간 프로세스·진행상황이 공유되지 않음 | 팀 차원의 자원 배분/우선순위 판단 불가 |
| 어떤 투자사가 어떤 매물에 관심 있는지 데이터화 안 됨 | 매칭이 직관에만 의존 |

### 1.3 목표 (To-Be)
1. 구주 세일즈 활동을 **구조화된 데이터로 축적**한다.
2. 팀 구성원이 **누가-무엇을-어디까지** 진행 중인지 한 화면에서 본다.
3. "매물 × 투자사"의 진척을 **파이프라인으로 관리**한다.
4. 축적된 데이터를 **대시보드**로 가시화해 의사결정을 돕는다.

### 1.4 성공 지표 (제품 도입 후 측정)
- 등록된 투자사/조합/컨택 이력 수의 지속 증가
- 매물당 평균 노출 투자사 수
- 단계별 전환율 및 평균 소요 기간(cycle time) 가시화
- 팀 내 "내가 모르던 컨택 이력을 확인했다"는 정성 피드백

## 2. 핵심 컨셉: B2B CRM 객체를 구주 세일즈에 매핑

이 제품의 설계 근간. Salesforce/Pipedrive/HubSpot의 객체 모델을 구주 세일즈 도메인에 대응시킴.

| 일반 B2B CRM | 본 제품 | 설명 |
|---|---|---|
| Account (고객사) | **투자사 (Investor)** | 구주 매수 후보 — VC/CVC/PEF/세컨더리 펀드 등 |
| — | **조합 (Fund)** | 투자사가 운용하는 펀드. 드라이파우더·주목적·만기 보유 |
| Contact (담당자) | **컨택 (Contact)** | 투자사 내 의사결정자/심사역 |
| Product (상품) | **매물 (Listing)** | 우리가 파는 포트폴리오사 구주 |
| Opportunity (영업기회) | **딜 (Deal)** | **매물 × 투자사**의 교차점 = 하나의 세일즈 건 |
| Activity (활동) | **활동/타임라인 (Activity)** | 컨택 이력 기록 |
| Pipeline Stage | **세일즈 단계** | 롱리스트 → … → 클로징 |

> **핵심 통찰**: "딜(Deal)"은 *어떤 매물을 어떤 투자사에 파는가*라는 다대다(N:M) 관계를 해소하는 객체다.
> 매물 1개는 여러 투자사에 노출되고, 투자사 1곳은 여러 매물을 검토한다. 모든 파이프라인·타임라인·대시보드는 이 Deal 객체를 중심으로 돈다.

## 3. 사용자 및 권한

| 페르소나 | 니즈 | 주 사용 기능 |
|---|---|---|
| **심사역 (담당자)** | 본인 딜 진행, 이력 기록, 네트워크 활용 | 딜 보드, 활동 기록, 투자사/조합 조회 |
| **팀 리드 / 대표** | 전체 진척·자원 배분 파악 | 대시보드, 전체 파이프라인 뷰 |

- **멀티유저 전제**: 데이터는 팀 전체가 공유(투자사·조합·컨택·매물은 공용 자산). 딜에는 `담당자(owner)`가 존재.
- 인증 필수, 외부 비공개. 권한 모델은 §8 참조.

## 5. 기능 명세

### 5.1 기능 우선순위 (Phase)
- **Phase 1 (MVP)**: 인증 · 핵심 객체 CRUD(투자사·조합·컨택·운용펀드·매물) · 딜 파이프라인(칸반) · 활동 타임라인
- **Phase 2**: 적합도 매칭 · 소개 네트워크 · 검색/필터 · 알림
- **Phase 3**: 대시보드/애널리틱스 · 외부 연동(메일/캘린더) · 데이터 임포트

---

### 5.2 Phase 1 — MVP

#### F1. 인증 / 팀
- 이메일 기반 로그인. 같은 조직 구성원만 접근.
- **수용 기준**: 비로그인 시 모든 데이터 접근 차단. 로그인 후 본인 정보 표시.

#### F2. 투자사 · 조합 · 컨택 관리
- 투자사 등록/수정/조회. 투자사 상세에서 하위 **조합 N개**, **컨택 N명**을 함께 관리.
- 조합에 드라이파우더·주목적·만기·섹터 등 입력.
- **수용 기준**: 투자사 상세 페이지에서 소속 조합·컨택·연결된 딜·활동 이력이 한 화면에 보임(Salesforce의 Related Lists 패턴).

#### F3. 매물(구주) 관리
- 포트폴리오사 구주를 매물로 등록. 회사명·상태 중심의 가벼운 정보 입력(보유 수량·취득가 등 cap table 데이터는 미취급).
- **운용펀드 태깅**: 매물 등록·수정 시 **1개 이상의 운용펀드**를 소속으로 연결(`ListingFund`). 같은 매물의 **중복 소속 허용**.
- 운용펀드 마스터(이름·만기일·상태)는 별도 간단 CRUD로 관리.
- **수용 기준**: 매물 목록을 상태/**운용펀드**로 필터. 매물 상세에서 (1) 소속 운용펀드 목록, (2) "이 매물을 검토 중인 투자사(=연결된 딜)" 리스트 표시.

#### F4. 딜 파이프라인 (칸반 보드) ⭐
- 세일즈 단계를 **칸반 컬럼**으로 표현, 딜 카드를 드래그로 단계 이동(Pipedrive 패턴).
- **단계 정의**:
  `1.롱리스트` → `2.컨택` → `3.미팅` → `4.자료검토` → `5.딜리전스/IR` → `6.협상/텀시트` → `7.클로징` → `Won/Lost`
- 카드에 표시: 매물명 · 투자사명 · 담당자 · 예상금액 · 다음 액션(+예정일).
- **수용 기준**: 드래그로 단계 변경 시 즉시 저장. `next_action_date`가 지난 카드는 시각적 경고. 매물별/담당자별 필터 제공.

#### F5. 활동 타임라인 (컨택 이력) ⭐
- 딜 또는 투자사 상세에서 활동을 **시간순 타임라인**으로 표시.
- 빠른 기록: 유형·일시·내용·컨택 선택 후 1클릭 저장(HubSpot의 quick-log 패턴).
- 활동 기록 시 해당 컨택의 `last_contacted_at` 자동 갱신.
- **수용 기준**: 누가(작성자) 언제 무엇을 기록했는지 모두 보임. 팀원이 남긴 이력도 동일하게 조회 가능.

#### F6. 매물–투자사 태깅(딜 생성)
- 매물 또는 투자사에서 상대를 선택해 딜 생성 = "이 매물을 이 투자사에 소개함" 태깅.
- **수용 기준**: 동일 매물×투자사 중복 딜 방지. 생성 즉시 파이프라인 `롱리스트`에 등장.

---

### 5.3 Phase 2 — 차별화 기능

#### F7. 적합도 매칭 (구주 ↔ 조합)
- 매물 1건 기준, 등록된 조합들에 대해 **적합도 점수**를 룰 기반으로 산출·정렬.
- 점수 인풋(예시 가중치):
  - `stage_focus`에 매물 stage 포함 (+)
  - `sector_focus`에 매물 sector 포함 (+)
  - `dry_powder` ≥ 예상 거래규모 (+)
  - `secondary_appetite` = 적극/가능 (+)
  - `maturity_date`가 임박(잔여 운용기간 부족) (−)
- **수용 기준**: "이 매물을 보여줄 만한 투자사 Top N"을 추천 리스트로 제시. 추천 → 딜 생성 1클릭.

#### F8. 소개 네트워크 추적
- `intro_path`를 구조화: 소개자, 관계, 일자. 투자사별 "어떻게 연결됐나" 히스토리.
- **수용 기준**: 특정 투자사로의 진입 경로를 한눈에 확인. (향후 네트워크 그래프 시각화 확장 여지)

#### F9. 검색 · 필터 · 저장된 뷰
- 전역 검색(투자사/매물/컨택). 리스트 뷰에 다중 필터 + 저장된 뷰(Attio/Airtable 패턴).
- **수용 기준**: "내 딜 중 협상단계", "긴급도 높은 매물" 등을 저장해 재사용.

#### F10. 알림 / 리마인더
- `next_action_date` 도래, 조합 만기 임박, 장기 미접촉 투자사 등을 대시보드 상단/알림으로 노출.
- **수용 기준**: 오늘 해야 할 액션이 로그인 시 가장 먼저 보임.

---

### 5.4 Phase 3 — 대시보드 및 연동

#### F11. 대시보드 ⭐
표시 위젯:
- **파이프라인 현황**: 단계별 딜 수 / 합산 예상금액 (펀넬)
- **매물별 진척**: 매물마다 노출 투자사 수 · 최고 도달 단계
- **운용펀드별 현황**: 펀드별 보유 매물 수 · 세일즈 진행 매물 수 · 만기까지 잔여기간
- **전환율 & cycle time**: 단계 간 전환율, 평균 소요일
- **활동량**: 심사역별 / 투자사별 활동 빈도
- **드랍 분석**: `lost_reason` 집계
- **수용 기준**: 기간 필터 + **운용펀드 필터**(펀드명 선택 시 해당 펀드 보유 매물·관련 딜만 표시). 위젯 클릭 시 해당 딜 리스트로 드릴다운.

#### F12. 외부 연동 (선택)
- 캘린더/메일 연동으로 미팅·메일을 활동으로 자동 수집. (난이도 높음, 후순위)

#### F13. 데이터 임포트
- 기존 스프레드시트(투자사 리스트, 조합 정보)를 CSV/엑셀로 일괄 업로드.
- **수용 기준**: 컬럼 매핑 후 투자사·조합 일괄 생성.

## 7. 기술 스택 (Claude Code 친화적 권장안)

> 비개발자가 Claude Code와 빌드하기 좋은, 멀티유저·인증·공유 데이터를 잘 지원하는 조합. 이미 다른 스택을 쓰고 있다면 알려주면 맞춰 조정.

| 레이어 | 권장 | 이유 |
|---|---|---|
| 프레임워크 | **Next.js (App Router) + TypeScript** | Claude Code 지원 성숙, 풀스택 단일 코드베이스 |
| UI | **Tailwind CSS + shadcn/ui** | 빠른 구성, 일관된 컴포넌트 |
| DB/인증/스토리지 | **Supabase** (Postgres + Auth + RLS + Storage) | 멀티유저 인증·행수준 보안·파일 저장 한번에 |
| 데이터 패칭 | **TanStack Query** | 서버 상태 관리 |
| 칸반 DnD | **dnd-kit** | F4 파이프라인 드래그 |
| 차트 | **Recharts** 또는 **Tremor** | F11 대시보드 |
| 배포 | **Vercel** | Next.js 무중단 배포 |

대안: DB를 단순화하려면 Prisma + Postgres 직접 구성도 가능하나, 팀 인증·권한이 필요하므로 **Supabase 권장**.

## 12. 디자인 가이드 (와이어프레임 & Stitch)

### 12.1 공통 셸
모든 메인 화면은 동일한 레이아웃 셸을 공유한다.
- 상단 바: 로고(SecondaryDesk) · 전역 검색 · 사용자 아바타
- 좌측 세로 네비게이션: 대시보드 / 딜 보드 / 투자사 / 매물 / 알림
- 본문: 화면별 콘텐츠

### 12.2 화면별 레이아웃 명세 (와이어프레임 기준)

**A. 딜 파이프라인 (칸반 보드)** — 메인 작업 공간
- 상단 필터 행: 매물·담당자·긴급도 드롭다운 + 우측 `+ 딜` 버튼
- 가로 단계 컬럼: 롱리스트 → 컨택 → 미팅 → 자료검토 → 딜리전스/IR → 협상/텀시트 → 클로징 (가로 스크롤)
- 컬럼 헤더: 단계명 + 딜 개수
- 딜 카드: 매물명(강조) · 투자사명 · 담당자 · 예상금액 · 다음 액션(+마감일)
- 마감 임박 카드는 빨간 점으로 시각 경고 (F4 + F10 연동)

**B. 투자사 상세** — Related Lists 패턴
- 헤더: 투자사명 · 유형 배지 · Tier 배지 · 담당자
- 탭: 개요 · 조합 · 컨택 · 딜 · 활동
- 본문 좌측: 조합 카드(조합명·드라이파우더·주목적·만기·섹터 칩·구주 선호도) + 컨택 목록(이름·직책·결정권 표시)
- 본문 우측: 세로 활동 타임라인(날짜·유형·요약) + `+ 활동 기록` 버튼

**C. 대시보드**
- 지표 카드 4개: 진행 중 딜 / 합산 예상금액 / 평균 cycle time / 이번주 활동
- 좌측: 파이프라인 펀넬(단계별 가로 막대)
- 우측: 오늘의 액션(마감 임박=빨강, 예정=주황, 미접촉=회색)
- 하단: 매물별 진척 테이블(매물 / 노출 투자사 수 / 최고 단계 / 전환율)

**D. 활동 기록 모달**
- 필드: 활동 유형 · 일시 · 대상 컨택 · 연결 딜 · 내용 · 첨부
- 하단: 취소 / 저장

### 12.3 비주얼 톤
- Linear·Attio 지향: 흰색 표면, 0.5px 얇은 구분선, 차분한 중립 그레이 배경
- 포인트 컬러: 인디고/블루 (정보), 빨강(마감/위험), 주황(주의)
- 모서리 라운드 8px, 그림자·그라데이션 최소화
- 데이터 밀도 높은 업무툴 — 가독성·빠른 입력 우선
- 라이트/다크 모드 지원, 데스크톱 우선

### 12.4 Google Stitch 프롬프트
> Stitch는 화면 단위로 생성. ①을 테마로 먼저 입력 후 ②~⑤를 화면별로 입력.

**① 테마/컨텍스트**
> SecondaryDesk — 벤처캐피탈 심사역이 보유 포트폴리오사의 구주(secondary shares)를 후속 투자사에 판매하는 영업 활동을 관리하는 내부용 B2B Sales CRM. 데이터 밀도가 높은 실무 도구로, Linear·Attio처럼 깔끔하고 절제된 톤. 좌측 세로 네비게이션(대시보드 · 딜 보드 · 투자사 · 매물 · 알림), 흰색 표면 + 얇은 구분선 + 차분한 중립 그레이 배경에 인디고/블루 포인트 컬러. 그림자·그라데이션 최소화, 모서리 라운드 8px. 한국어 UI, 데스크톱 우선, 라이트/다크 모드.

**② 딜 파이프라인 (칸반 보드)**
> 딜 파이프라인 칸반 보드 화면. 좌측 네비게이션과 상단 검색바 포함. 상단에 필터 칩(매물·담당자·긴급도)과 우측 '+ 딜' 버튼. 가로로 배치된 단계 컬럼: 롱리스트 → 컨택 → 미팅 → 자료검토 → 딜리전스/IR → 협상/텀시트 → 클로징. 각 컬럼 헤더에 단계명과 딜 개수. 각 딜 카드에는 매물명(굵게), 투자사명, 담당자 아바타, 예상 거래금액, '다음 액션 + 마감일' 표시. 마감 임박 카드는 빨간 점으로 강조. 드래그 가능한 카드 느낌.

**③ 투자사 상세**
> 투자사 상세 페이지. 상단에 투자사명, 유형 배지(VC), Tier 배지, 담당자. 그 아래 탭: 개요 · 조합 · 컨택 · 딜 · 활동. 본문 좌측 넓은 영역: '조합' 카드 목록(조합명, 드라이파우더, 주목적, 만기일, 섹터 태그 칩, 구주 선호도 배지)과 '컨택' 목록(이름, 직책, 의사결정권자 별표). 본문 우측 좁은 영역: 세로 활동 타임라인(날짜·유형·한 줄 요약, 미팅/통화/자료발송/노트)과 하단에 '+ 활동 기록' 버튼.

**④ 대시보드**
> 분석 대시보드 화면. 상단에 기간 필터. 지표 카드 4개 한 줄: 진행 중 딜, 합산 예상금액, 평균 cycle time, 이번주 활동. 좌측 큰 카드: 파이프라인 펀넬(단계별 가로 막대, 위에서 아래로 줄어드는 형태). 우측 카드: '오늘의 액션' 리스트(마감 임박은 빨강, 예정은 주황, 미접촉은 회색 점). 하단 전체 폭 테이블: 매물별 진척(매물 / 노출 투자사 수 / 최고 단계 / 전환율).

**⑤ 활동 기록 모달**
> 활동 기록 입력 모달. 필드: 활동 유형(미팅·통화·이메일·메신저·자료발송·IR·노트 셀렉트), 일시, 대상 컨택 선택, 연결된 딜 선택, 내용 텍스트영역, 첨부파일. 하단에 '취소'·'저장' 버튼. 어둡게 흐려진 배경 위 중앙 카드 형태.

### 12.5 Stitch 디자인 레퍼런스 사용 지침
> Stitch 시안은 **시각 레퍼런스**로만 사용한다. 생성된 코드를 앱 소스로 복사하지 않는다(목업 데이터·단독 Tailwind 마크업이라 §4 스키마·라우팅·shadcn/ui 체계와 충돌).

**리포지토리 배치**
- `design-reference/` : Stitch `.zip`(또는 React 앱) 내보내기 결과. 참고용, 빌드 대상 아님.
- `docs/wireframes/` : 화면별 스크린샷 PNG. 이 문서(§12.2)와 함께 목표 룩앤필의 기준.
- Stitch 프로젝트 공유 링크: https://stitch.withgoogle.com/projects/2672674626820083439 (라이브 시안 확인용)

**Claude Code 지시 템플릿**
```
design-reference/ 폴더와 PRD.md §12를 시각 레퍼런스로 삼되, 코드는 복사하지 말 것.
PRD §4 스키마와 shadcn/ui 컴포넌트로 재구현하고, Stitch 목업의 색·간격·레이아웃만 맞춘다.
목업의 더미 데이터·하드코딩 값은 가져오지 말고 실제 데이터 연결 구조로 구현한다.
```

**원칙**
- 레퍼런스(Stitch) > 복사: 레이아웃·여백·색상 톤만 차용, 컴포넌트는 shadcn/ui로 새로 조립.
- Stitch 화면명과 PRD 화면명을 일치시켜 추적성 유지(딜 보드 / 투자사 상세 / 대시보드 / 활동 모달).
- 디자인 토큰(색·라운드·폰트)은 §12.3을 단일 출처로 삼고, 시안과 PRD가 어긋나면 PRD를 갱신해 일치시킨다.
